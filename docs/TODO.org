# Core Engineering Principles

You must strictly abide by these principles when generating any file:

 1. **Don't Repeat Yourself (DRY):** Hardcoded secrets or environment-specific
    paths are forbidden. Centralize variables using `.env.template` files (using `change_me_to_<identifier>` placeholders). Use `Makefile` structures to pass these variables natively into local components.

 2. **Immutable Infrastructure:** You are strictly forbidden from using
    floating tags (e.g., `latest`, `main`). All external software, Docker images, and dependencies must be pinned to explicit, static version tags.

 3. **Principle of Least Privilege:** Do not run containers as root if
    avoidable. Provision isolated database users with strict connection and memory limits tailored to their specific workload (e.g., vector sorting vs. proxy logging).

 4. **Idempotency:** Initialization scripts (`init.sql`, bash scripts) must
    be safely repeatable without causing destructive state changes or failures.

 5. **State Segregation:** Ephemeral container data must be strictly
    separated from persistent volumes, with clear variables dictating external drive paths for heavy I/O operations.

I use a custom Python parsing script (`newcode.py`) to extract your generated
code into my local directory hierarchy. You must output all code,
configurations, or script modifications within a single Markdown code block.

 * Each distinct file must be separated by the exact syntax
   `====> <filename> <====` on its own line immediately preceding the file
   content.

 * The `<filename>` MUST be a path relative to the `openclaw` directory that
   strictly begins with `./`
   (e.g., `====> ./litellm-proxy/docker-compose.yml <====`).
   Do not provide filenames without this format.

 * **Directory State Management:** The file `./struct.txt` acts as the
